### 🔍 기술 리소스 및 병목 지점 분석

MVP 출시 목표 달성을 위해 현재 상태에서 예상되는 주요 리소스 요구사항과 잠재적 병목 지점을 다음과 같이 분석했습니다.

#### 1. 백엔드 (FastAPI / SQLAlchemy) 병목 지점

*   **데이터 모델의 복잡성:** `models.py`에 정의된 데이터 모델(`roadmap_plans`, `qa_sessions` 등)이 Pro Tier 요구사항(맞춤형 로드맵, AI Q&A 기록)을 충분히 담고 있는지 확인해야 합니다. 만약 AI 관련 기록이나 로드맵 생성 과정에서 필요한 중간 상태 필드가 누락된다면, 추후 기능 추가 시 대규모 마이그레이션이 발생할 수 있습니다.
    *   **검증 필요:** 현재 데이터 모델(`models.py`)이 **Pro Tier의 모든 기록 및 계산을 위한 최소한의 정합성**을 확보했는지 확인해야 합니다.
*   **AI/로직 통합 지점:** 만약 선택된 기능(로드맵 생성 또는 Q&A)에 AI 모델을 연동한다면, 이 로직이 FastAPI 엔드포인트 내에서 **지연 시간(Latency)**을 유발하지 않도록 비동기 처리(`async`/`await`)와 적절한 캐싱 전략을 설계해야 합니다. 이는 MVP 출시 속도에 가장 큰 영향을 미치는 병목입니다.
*   **API 성능:** 단순 CRUD를 넘어선 로드맵 생성과 같은 복잡한 데이터 집계 요청 시, SQLAlchemy 쿼리가 비효율적으로 작성되어 DB 부하가 발생할 가능성이 있습니다. 초기에는 단순 조회에 집중하고, 복잡한 연산은 별도의 비동기 작업으로 분리하는 구조를 선행해야 합니다.

#### 2. 프론트엔드 (React / Tailwind) 병목 지점

*   **컴포넌트의 재사용성 및 상태 관리:** `TrustScoreGauge` 컴포넌트는 디자인 사양을 완벽히 반영해야 하며, 이 컴포넌트가 다른 페이지에서도 일관되게 작동하도록 **상태(State)**와 **Props** 관리가 명확해야 합니다. 만약 로드맵이나 Q&A 기록이 추가될 경우, 이를 관리할 전역 상태(예: Zustand, Redux) 설계가 MVP 단계에서 너무 복잡해지지 않도록 초기부터 단순하게 설계해야 합니다.
*   **디자인 정합성 유지:** Lead Designer가 확정한 디자인 사양을 픽셀 단위로 정확히 구현하는 데 시간이 소요될 수 있습니다. Tailwind CSS를 사용하더라도, 미세한 간격(Spacing)이나 색상(Color Palette)의 일관성을 유지하기 위한 **Design Token** 시스템 구축이 필요합니다.
*   **데이터 바인딩:** 백엔드에서 제공하는 데이터가 프론트엔드의 시각화 요구사항과 정확히 매핑되도록 데이터 흐름을 초기 단계부터 명확히 정의해야 합니다.

#### 3. 통합 및 배포 병목 지점

*   **CI/CD 설정 부재:** MVP 출시를 위해서는 코드를 빌드하고 배포하는 자동화된 파이프라인(CI/CD)이 필수적입니다. 현재 상태에서는 수동 배포에 의존할 경우, 작은 변경에도 오류가 발생하기 쉬우며 이는 곧 출시 지연으로 이어집니다.
*   **환경 분리:** 개발, 스테이징, 프로덕션 환경 간의 설정(API 키, DB 연결 문자열 등)을 안전하게 분리하고 관리하는 과정이 누락되면 배포 시점에 치명적인 오류가 발생합니다.

### 2. Trust Score Gauge 컴포넌트 상세 요구사항 (UI/UX Specification)

`TrustScoreGauge` 컴포넌트는 단순한 게이지 표시를 넘어, 사용자의 성취감을 극대화하는 경험을 제공해야 합니다.

#### A. 레이아웃 및 구조 (Layout & Structure)
*   **목표:** 시각적으로 신뢰도(Trust)와 성과(Success)의 연결 고리를 명확히 제시한다.
*   **구조:** 전체 컴포넌트는 3단계 레이어로 구성되어야 합니다.
    1.  **Header (Score Display):** 현재 점수(`X/100`)를 가장 크고 명확하게 표시합니다.
    2.  **Gauge Visualization:** 실제 점수 진행 상태를 시각적으로 표현하는 핵심 게이지(Arc 또는 Ring 형태)입니다.
    3.  **Contextual Feedback (Status Indicator):** 현재 점수가 어떤 의미인지 설명하는 텍스트 피드백을 제공합니다.

#### B. 시각적 구현 상세 (Visual Implementation Details)
*   **Gauge Visualization:** 원형 게이지(Circular Progress Bar)를 채택하며, **Deep Ocean Blue**로 배경을 설정하고, 실제 진행 상태는 **Aqua Teal** 색상으로 표현하여 대비 효과를 극대화합니다.
*   **Color Mapping:** 점수가 낮을수록 (예: 0-30%)은 연한 파란색 계열을 사용하고, 고득점일수록 (예: 80-100%)은 깊은 네이비/블루 색상을 사용하여 시각적 무게감을 부여합니다.
*   **Typography:** 점수(`X/100`)는 **Poppins Bold**로, 나머지 설명 텍스트(예: "당신의 현재 신뢰도")는 **Inter Regular**로 처리하여 계층 구조를 명확히 합니다.

| :--- | :--- | :--- |
| `score` (0-100) | 게이지의 진행률 계산의 핵심 값 | 필수 |
| `status_level` (String) | 점수에 따른 감정적 상태 분류 (예: Low, Medium, High, Expert) | Contextual Feedback 텍스트 결정에 사용 |
| `feedback_message` (String) | 사용자에게 제공될 구체적인 피드백 문구 | '성공 경험' 메시지 연관성 확보 |

app = FastAPI(title="Trust Score Gauge API")

# --- Mock Data/Logic for Demonstration ---
def calculate_score(plan: models.RoadmapPlan, qas: models.QASession) -> float:
    """
    가상의 Trust Score 계산 로직. 실제 비즈니스 로직으로 대체되어야 함.
    예시: 계획 진행률과 Q&A 참여도를 기반으로 점수 산출.
    """
    progress_weight = 0.6  # 진행률 가중치
    qa_weight = 0.4       # Q&A 참여도 가중치

    # 진행률은 0.0에서 1.0 사이 값이라고 가정
    progress = float(plan.status.replace('In Progress', '75%').replace('Completed', '100%')) / 100.0 if plan.status in ['In Progress', 'Completed'] else 0.0
    qa_completion = 1.0 if qas else 0.0

    score = (progress * progress_weight) + (qa_completion * qa_weight)
    return round(score, 2)

# --- API Endpoints ---

@app.post("/trustscore/calculate", response_model=models.TrustScoreGauge)
def calculate_and_save_trust_score(plan_id: int, qas_id: int, db: Session = Depends(get_db)):
    """
    특정 계획과 Q&A 세션을 기반으로 Trust Score를 계산하고 저장합니다.
    """
    plan = db.query(models.RoadmapPlan).filter(models.RoadmapPlan.id == plan_id).first()
    qa = db.query(models.QASession).filter(models.QASession.id == qas_id).first()

    if not plan:
        raise HTTPException(status_code=404, detail="Roadmap Plan not found")
    if not qa:
        # Q&A 기록이 없더라도 점수 계산은 가능하도록 처리 (기본값 설정)
        qa = models.QASession(plan_id=plan_id, question="", answer="")

    # 1. 점수 계산 로직 실행
    calculated_score = calculate_score(plan, qa)

    # 2. TrustScoreGauge 저장 또는 업데이트
    trust_score_record = db.query(models.TrustScoreGauge).filter(models.TrustScoreGauge.user_id == plan.user_id).first()

    if trust_score_record:
        trust_score_record.score = calculated_score
        trust_score_record.updated_at = datetime.utcnow()
    else:
        new_score = models.TrustScoreGauge(user_id=plan.user_id, score=calculated_score)
        db.add(new_score)

    db.commit()
    db.refresh(trust_score_record if trust_score_record else new_score) # 새로 생성된 객체 갱신 반영

    return {"message": "Trust Score calculated and saved successfully", "score": trust_score_record.score}


@app.get("/trustscore/{user_id}", response_model=models.TrustScoreGauge)
def get_user_trust_score(user_id: int, db: Session = Depends(get_db)):
    """
    특정 사용자의 Trust Score를 조회합니다.
    """
    trust_score = db.query(models.TrustScoreGauge).filter(models.TrustScoreGauge.user_id == user_id).first()
    if not trust_score:
        raise HTTPException(status_code=404, detail="User Trust Score not found")
    return trust_score

### B. 기본 기능 (Must-Have - Free Tier Focus)
1.  **Trust Score Gauge:** 사용자가 입력한 정보를 기반으로 실시간 신뢰도 점수를 시각화한다.

def calculate_trust_score(qa_sessions: list) -> float:
    """QA 세션 기록을 기반으로 전체 신뢰 점수를 계산합니다."""
    total_score = 0.0
    if not qa_sessions:
        return 0.0
    for session in qa_sessions:
        # 예시 로직: 답변의 품질과 관련성을 기반으로 점수 기여도 합산
        session_score = session.score_contribution
        total_score += session_score
    
    # 최대 점수를 100으로 제한하고 정규화 (실제 비즈니스 로직에 따라 수정 필요)
    final_score = min(100.0, total_score * 1.5) 
    return round(final_score, 2)

        new_session = QA_Session(**session_data)
        db.add(new_session)
        db.commit()
        db.refresh(new_session)
        return new_session

from sqlalchemy import Column, Integer, Float, DateTime, String
from sqlalchemy.ext.declarative import declarative_base
from datetime import datetime

Base = declarative_base()

class RoadmapPlan(Base):
    __tablename__ = "roadmap_plans"
    id = Column(Integer, primary_key=True, index=True)
    user_id = Column(Integer, index=True)  # 사용자 ID (외래 키)

    status = Column(String)             # 예: 'Draft', 'InProgress', 'Completed'
    created_at = Column(DateTime, default=datetime.utcnow)

class QASession(Base):
    __tablename__ = "qa_sessions"
    id = Column(Integer, primary_key=True, index=True)
    plan_id = Column(Integer, index=True)  # RoadmapPlan ID (외래 키)
    question = Column(String)
    answer = Column(String)
    created_at = Column(DateTime, default=datetime.utcnow)

class UserTrustScore(Base):
    __tablename__ = "user_trust_scores"
    id = Column(Integer, primary_key=True, index=True)
    user_id = Column(Integer, unique=True, index=True)  # 사용자 ID (외래 키)
    trust_score = Column(Float, default=0.0)           # 최종 신뢰 점수
